## Design Philosophy

### Core Principles

| Principle | Description |
|-----------|-------------|
| **Dark-First** | Professional dark theme optimized for extended use; reduces eye strain |
| **Minimal Cognitive Load** | Clear visual hierarchy; users find what they need instantly |
| **Consistent Patterns** | Predictable interactions across all components |
| **Accessibility Native** | WCAG AA compliance baked into every design decision |
| **Performance Conscious** | Lightweight components; no unnecessary visual complexity |

## Design Tokens

### Color Palette

#### Background Colors

| Token | Hex | RGB | Usage |
|-------|-----|-----|-------|
| `--bg-primary` | `#0a0a0b` | `10, 10, 11` | Main app background |
| `--bg-secondary` | `#141416` | `20, 20, 22` | Cards, elevated surfaces |
| `--bg-tertiary` | `#1c1c1f` | `28, 28, 31` | Hover states, subtle elevation |
| `--bg-elevated` | `#232326` | `35, 35, 38` | Modals, dropdowns, popovers |

#### Text Colors

| Token | Hex | Contrast Ratio | Usage |
|-------|-----|----------------|-------|
| `--text-primary` | `#fafafa` | 19.2:1 | Primary text, headings |
| `--text-secondary` | `#a1a1aa` | 7.1:1 | Secondary text, labels |
| `--text-muted` | `#71717a` | 4.6:1 | Placeholders, disabled (AA compliant) |
| `--text-inverse` | `#0a0a0b` | N/A | Text on light/accent backgrounds |

#### Accent Colors

| Token | Hex | Usage | Accessible Pair |
|-------|-----|-------|-----------------|
| `--accent-primary` | `#3b82f6` | Primary actions, links, focus rings | White text (4.5:1) |
| `--accent-primary-hover` | `#2563eb` | Primary hover state | White text (5.2:1) |
| `--accent-success` | `#22c55e` | Success states, completed tasks | Dark text (4.7:1) |
| `--accent-warning` | `#f59e0b` | Warnings, medium priority | Dark text (4.9:1) |
| `--accent-danger` | `#ef4444` | Errors, delete actions, high priority | White text (4.5:1) |
| `--accent-info` | `#06b6d4` | Informational, low priority | Dark text (4.6:1) |

#### Priority Colors (Task-Specific)

| Priority | Color | Token | Badge Style |
|----------|-------|-------|-------------|
| High | `#ef4444` | `--priority-high` | Solid red background |
| Medium | `#f59e0b` | `--priority-medium` | Solid amber background |
| Low | `#06b6d4` | `--priority-low` | Solid cyan background |
| None | `#71717a` | `--priority-none` | Muted gray outline |

#### Border Colors

| Token | Hex | Usage |
|-------|-----|-------|
| `--border-default` | `#27272a` | Default borders |
| `--border-subtle` | `#1f1f22` | Subtle separators |
| `--border-focus` | `#3b82f6` | Focus rings (2px) |

#### Button States

| State | Visual Change |
|-------|---------------|
| Default | Base colors as defined |
| Hover | Background darkens/lightens, shadow-md |
| Active | Scale 0.98, background darker |

| Disabled | Opacity 0.5, cursor not-allowed |
| Loading | Spinner icon, text hidden or "Loading..." |

## Accessibility (WCAG AA)

### Color Contrast Requirements

| Element | Minimum Ratio | Our Implementation |
|---------|--------------|-------------------|
| Normal text | 4.5:1 | `--text-primary` on `--bg-primary` = 19.2:1 |
| Large text (18px+) | 3:1 | All headings exceed 7:1 |
| UI components | 3:1 | All interactive elements exceed 4.5:1 |
| Focus indicators | 3:1 | `--border-focus` on any background = 5.2:1 |

### Keyboard Navigation

#### Focus Order

1. Skip to main content link (visually hidden until focused)
2. Header navigation (logo, search, notifications, user menu)
3. Tab navigation (All, Active, Completed)
4. Filter controls
5. Task list items (in order)
6. Add task button
7. Footer links

#### Focus Indicators

```css
/* Visible focus ring for all interactive elements */
.focus-visible {
  @apply outline-none ring-2 ring-accent ring-offset-2;
  @apply ring-offset-background-primary;
}

/* Keyboard-only focus (hide on mouse click) */
:focus:not(:focus-visible) {
  @apply outline-none ring-0;
}
```

## Animation Guidelines

### Principles

1. **Purposeful**: Animations guide attention and provide feedback
2. **Subtle**: Never distracting or excessive (150-300ms duration)
3. **Interruptible**: Users can continue interacting during animations
4. **Respectful**: Honor `prefers-reduced-motion` settings

### Animation Tokens

| Token | Duration | Easing | Usage |
|-------|----------|--------|-------|
| `--duration-fast` | 150ms | `ease-out` | Hovers, micro-interactions |
| `--duration-normal` | 200ms | `ease-in-out` | Transitions, reveals |
| `--duration-slow` | 300ms | `ease-in-out` | Modals, page transitions |
| `--duration-slower` | 500ms | `spring` | Complex animations (Framer Motion) |

### Component Animations (Framer Motion)

```jsx
// Task card enter animation
```

## User Scenarios & Testing

### User Story 1 - View and Manage Tasks (Priority: P1)

A user opens the dashboard to see their tasks organized by due date, with clear visual indicators for priority and completion status.

**Why this priority**: Core functionality - without task visibility, the app has no value.

**Independent Test**: Can be fully tested by loading the dashboard with sample tasks and verifying all visual elements render correctly.

**Acceptance Scenarios**:

1. **Given** a user is logged in with existing tasks, **When** they view the dashboard, **Then** they see tasks grouped by date with priority badges, due dates, and checkboxes visible.
2. **Given** a user hovers over a task card, **When** the card receives focus, **Then** it displays a subtle border color change and shadow to indicate interactivity.
3. **Given** a user clicks a task checkbox, **When** the task is marked complete, **Then** the checkbox shows a checkmark, the title gets a strikethrough, and the card appearance changes to muted.

---

### User Story 2 - Create New Task via Modal (Priority: P1)

A user clicks the "Add Task" button, fills out the task form in a modal, and submits to create a new task.

**Why this priority**: Task creation is essential for any todo app to provide value.

**Independent Test**: Can be fully tested by clicking Add Task, filling the form, and verifying the task appears in the list.

**Acceptance Scenarios**:

1. **Given** a user clicks "Add Task", **When** the modal opens, **Then** it appears with a smooth fade-in animation, the title input is focused, and the close button is visible.
2. **Given** a user fills out the task form with title and priority, **When** they click "Create Task", **Then** the modal closes, a success toast appears, and the new task is visible in the list.
3. **Given** a user opens the modal on mobile, **When** the modal renders, **Then** it appears as a full-screen bottom sheet with adequate touch targets.

---

### User Story 3 - Filter and Search Tasks (Priority: P2)

A user uses the filter bar to search for specific tasks by keyword or filter by priority/status.

**Why this priority**: Enhances usability once core functionality is in place.

**Independent Test**: Can be tested by entering search text and selecting filters, then verifying the task list updates correctly.

**Acceptance Scenarios**:

1. **Given** a user types in the search field, **When** they enter "groceries", **Then** only tasks containing "groceries" are displayed with the matching text highlighted.
2. **Given** a user selects "High" from the priority filter, **When** the filter is applied, **Then** only high-priority tasks are shown and a filter chip appears indicating the active filter.
3. **Given** a user clicks the "Ã—" on a filter chip, **When** the filter is removed, **Then** the task list updates to show all tasks matching remaining filters.

---

### User Story 4 - Responsive Layout Adaptation (Priority: P2)

A user accesses the app on different devices (mobile, tablet, desktop) and the layout adapts appropriately.

**Why this priority**: Professional presentation across devices is essential for judges and real users.

**Independent Test**: Can be tested by resizing the browser window across breakpoints and verifying layout changes.

**Acceptance Scenarios**:

1. **Given** a user views the dashboard on mobile (<640px), **When** the page loads, **Then** the header shows a hamburger menu, task cards are stacked vertically, and the filter bar collapses to an icon.
2. **Given** a user views the dashboard on desktop (>1024px), **When** the page loads, **Then** the full navigation is visible, filter controls are inline, and task cards show all information in a single row.

---

### User Story 5 - Keyboard Navigation (Priority: P2)

A user navigates the entire application using only the keyboard without a mouse.

**Why this priority**: WCAG AA compliance and accessibility for users with motor disabilities.

**Independent Test**: Can be tested by navigating through the app using Tab, Enter, Escape, and Arrow keys only.

**Acceptance Scenarios**:

1. **Given** a user presses Tab repeatedly, **When** focus moves through the page, **Then** each interactive element receives a visible focus ring and the focus order follows the logical reading order.
2. **Given** a user focuses on the "Add Task" button and presses Enter, **When** the modal opens, **Then** focus moves to the first form field inside the modal.
3. **Given** a user presses Escape while a modal is open, **When** the modal closes, **Then** focus returns to the button that opened the modal.

---

### Edge Cases

- What happens when a task title is extremely long (200+ characters)?
  - Title truncates with ellipsis, full title visible on hover/focus via tooltip
- What happens when there are no tasks matching the current filter?
  - Empty state displays with message "No tasks match your filters" and option to clear filters
- What happens when the user's session expires while viewing tasks?
  - Toast notification appears, redirects to login with return URL preserved
- How does the system handle rapid checkbox toggling?
  - Debounced updates (300ms), optimistic UI update with rollback on failure

## Requirements

### Functional Requirements

- **FR-001**: System MUST render all UI components with specified color tokens and contrast ratios meeting WCAG AA (4.5:1 for normal text, 3:1 for large text)
- **FR-002**: System MUST display visible focus indicators (2px ring, --border-focus color) on all interactive elements when navigated via keyboard
- **FR-003**: System MUST adapt layout responsively at defined breakpoints (640px, 1024px)
- **FR-004**: System MUST provide all button variants (primary, secondary, ghost, destructive, link) with specified states (default, hover, active, focus, disabled, loading)
- **FR-005**: Task cards MUST display checkbox, title, priority badge, category tag, due date, and action buttons as specified
- **FR-006**: Modals MUST appear with overlay, be dismissible via close button or Escape key, and trap focus within
- **FR-007**: Toast notifications MUST appear in bottom-right corner with appropriate severity styling and auto-dismiss behavior
- **FR-008**: All form inputs MUST display appropriate states (default, focused, error, disabled) with accessible labels
- **FR-009**: System MUST render empty states with icon, heading, description, and call-to-action button when no content exists
- **FR-010**: System MUST respect user's `prefers-reduced-motion` preference by disabling animations

### Key Entities

- **Design Token**: Color, spacing, typography, shadow, or radius value used consistently across components
- **Component**: Reusable UI element with defined variants, sizes, and states
- **Page Layout**: Complete screen design showing component arrangement and responsive behavior
- **Interaction State**: Visual representation of a component in a specific state (hover, focus, active, disabled, loading)
